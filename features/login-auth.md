# Feature: Login Page and Supabase Auth

## Overview
Add a simple login gate to the app using Supabase Auth. A new `login.html` page handles email/password authentication. All existing pages (`index.html`, `statistikk.html`) redirect unauthenticated users to the login page. Supabase RLS policies are tightened so data is only accessible to authenticated users — not the public.

## How It Works

```
User visits index.html
    ↓
Check for valid Supabase session
    ↓ (no session)            ↓ (session valid)
Redirect to login.html     Load map normally
    ↓
User enters email + password
    ↓
Supabase Auth validates credentials
    ↓ (success)              ↓ (failure)
Redirect to index.html    Show error message
```

## User Management
Users are created **manually** in the Supabase dashboard — there is no self-registration.
- Go to Supabase → Authentication → Users → Invite user
- Create one account per team member
- To remove access, delete the user from the dashboard

## Implementation Tasks

### 1. Update RLS Policies
Replace the current public access policies with authenticated-only policies. Run this in the Supabase SQL editor:

```sql
-- markers table
DROP POLICY IF EXISTS "Public read" ON markers;
DROP POLICY IF EXISTS "Public update" ON markers;
DROP POLICY IF EXISTS "Public insert" ON markers;

CREATE POLICY "Auth read" ON markers FOR SELECT USING (auth.role() = 'authenticated');
CREATE POLICY "Auth update" ON markers FOR UPDATE USING (auth.role() = 'authenticated');
CREATE POLICY "Auth insert" ON markers FOR INSERT WITH CHECK (auth.role() = 'authenticated');

-- fellinger table
DROP POLICY IF EXISTS "Public read" ON fellinger;
DROP POLICY IF EXISTS "Public update" ON fellinger;
DROP POLICY IF EXISTS "Public insert" ON fellinger;

CREATE POLICY "Auth read" ON fellinger FOR SELECT USING (auth.role() = 'authenticated');
CREATE POLICY "Auth update" ON fellinger FOR UPDATE USING (auth.role() = 'authenticated');
CREATE POLICY "Auth insert" ON fellinger FOR INSERT WITH CHECK (auth.role() = 'authenticated');
```

### 2. Create `public/login.html`
- Simple centered login form with email and password fields
- A submit button labelled "Logg inn"
- An error message area (hidden by default, shown on failed login)
- A loading state on the button while the auth request is in progress
- No registration link — this is invite-only
- Style consistent with the existing app

Login logic:
```javascript
const { data, error } = await supabase.auth.signInWithPassword({
  email: emailInput.value,
  password: passwordInput.value
})

if (error) {
  // Show error message: "Feil e-post eller passord"
} else {
  window.location.href = 'index.html'
}
```

### 3. Auth Guard (`public/auth.js`)
Create a shared `auth.js` module that all protected pages load:
```javascript
async function requireAuth() {
  const { data: { session } } = await supabase.auth.getSession()
  if (!session) {
    window.location.href = 'login.html'
  }
  return session
}
```

### 4. Update `public/index.html`
- Add `<script src="auth.js"></script>` before map initialisation
- Call `await requireAuth()` as the first thing in the page's init function
- Add a logout button in the map UI (top right corner):
  ```javascript
  await supabase.auth.signOut()
  window.location.href = 'login.html'
  ```

### 5. Update `public/statistikk.html`
- Add `<script src="auth.js"></script>` before page initialisation
- Call `await requireAuth()` as the first thing in the page's init function

### 6. Update `supabaseClient.js`
- No changes needed to the client itself — Supabase automatically attaches the session token to all requests once the user is signed in

### 7. Update GitHub Actions (`deploy.yml`)
- No changes needed — `config.js` is already generated with the correct credentials

## Acceptance Criteria
- [ ] Visiting `index.html` without a session redirects to `login.html`
- [ ] Visiting `statistikk.html` without a session redirects to `login.html`
- [ ] Valid credentials log the user in and redirect to `index.html`
- [ ] Invalid credentials show a Norwegian error message
- [ ] Logout button signs the user out and redirects to `login.html`
- [ ] Unauthenticated requests to Supabase are rejected (verify in Supabase logs)
- [ ] `login.html` is accessible without a session (no auth guard on this page)

## Notes
- Run the RLS SQL **before** deploying the frontend changes — otherwise authenticated users will lose access during the transition window
- Test login locally with `config.local.js` before pushing to GitHub Pages
- Supabase sessions persist in the browser — users will stay logged in across page refreshes until they explicitly log out or the token expires (default 1 hour, configurable in Supabase → Auth → Settings)
- If a team member forgets their password, reset it via Supabase dashboard → Authentication → Users
- Do not add a "forgot password" flow to the login page — keep it simple and manage passwords manually via the dashboard
